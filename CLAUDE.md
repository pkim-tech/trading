## Project Overview
A z-score mean reversion backtesting and optimization system targeting leveraged ETFs. Runs parallel grid sweeps over take profit, stop loss, and hold time parameters to find optimal strategy configurations with positive alpha vs SPY. Now in live trading phase with manual execution via Schwab.

## Key Files
- `app.py` тАФ Streamlit UI for configuring and launching optimization sweeps
- `run_optimization_sweep.py` тАФ main parallel optimization engine (brute force + generational refinement)
- `backtester.py` тАФ core backtest simulation logic (`run_backtest`)
- `strategies.py` тАФ strategy classes: `ZScoreBreakout`, `TrailingExitZScoreBreakout`, `LimitOrderZScoreBreakout`, `LimitOrderTrailingExit`, `TrailingBuyZScoreBreakout`, `TrailingBothZScoreBreakout` (live default), `LimitExitZScoreBreakout`, `TrendFilteredZScore`
- `active_signals.py` тАФ live trading daemon: polls signal windows, manages `open_positions`/`watch_list`, fires Slack alerts. Delegates to (and re-exports, for backward compat): `signals_config.py` (config/tokens/Bolt app singleton), `signals_db.py` (DB layer), `signals_compute.py` (signal computation), `signals_charts.py` (chart PNGs), `signals_blocks.py` (Slack message posting/block builders), `signals_helpers.py` (small shared helpers), `signals_handlers.py` (Bolt interactive button/modal handlers), `signals_notify.py` (notify_*/reminder loops/reference-table/report тАФ the core, formerly the whole Slack-facing layer, split out 2026-07-14 for length).
- `data_collector.py` тАФ fetches and caches hourly price data
- `data_manager.py` тАФ data fetching and cache management
- `config.json` тАФ runtime config: tickers, hyperparameters, strategy selection
- `pages/` тАФ Streamlit multipage app views (Spatial Topology, Node Inspector)
- `scripts/daemon_status.py` тАФ checks whether `active_signals.py` is running and whether its process-start time is older than the newest edit among the live-trading source files (i.e. stale vs. already restarted) тАФ run instead of manually comparing `ps`/mtimes whenever a restart is pending.
- `scripts/run_v4_backfill_sweep.sh` тАФ v4 kernel-correctness backfill wrapper (fill-optimism resolution bounds, `fixed_sl` sweep, Open-check entry timing тАФ see `docs/backlog_cache.md`). `backtester._simulate_trail_both` now computes three parallel trailing-buy bounce-fill resolutions per node, since none of hourly OHLC proves the true intrabar path: **possible** (existing/unchanged, assumes Low-before-High), **pessimistic** (new, mirror-image Low-after-High assumption тАФ always same-bar-or-later and same-or-worse price than `possible`, a real bracket partner), **certain** (new, only resolves a fill when provable regardless of ordering тАФ can occasionally beat `possible` since deferring lets `running_low` fall further first, so it is *not* a pessimistic-price bound, just a no-guessing one). Island search and cliff-safety rank/filter on `MIN(possible, pessimistic, certain)` (`run_optimization_sweep.ROBUST_ALPHA_SQL`), not `possible` alone, so a node is only selected when it holds up under every resolution. Now loops 14 `stop_loss` values (densified 2026-07-15: `1,2,3,4,5,6,9,12,15,18,21,24,27,30`) x 2 `entry_timing` values through `run_optimization_sweep.py --version v4 --entry-timing <close|open_check> --max-phase <1|2|2.5|3>`, each a phase1тЖТ2тЖТ2.5(тЖТ3) run. Unlike `run_v3_backfill_sweep.sh`, every campaign shares one version string (`v4`) тАФ the real `stop_loss`/`entry_timing` `backtest_cache` columns disambiguate campaigns instead, so island/cliff-safety queries filter on those columns too (`run_optimization_sweep.py::_campaign_scope_sql`). New `sl_sweep_summary` table holds one rollup row per completed campaign for slice/trend queries without re-deriving island stats from raw `backtest_cache`. `--max-phase` (added 2026-07-15) caps a run at the given phase тАФ added after confirming Phase 3 held the best `MIN(possible,pessimistic,certain)` node in 0/30 tagged SOXL+KORU campaigns, and that island/cliff-safety selection only ever reads Phase1+2+2.5 data; `run_phase3_full` now also logs a `Phase3 best=... (pre-Phase3 best=..., IMPROVED/no improvement)` line whenever Phase 3 does run, for live confirmation. New `generation` column (nullable, `Phase2-Island` rows only) records which 1-indexed island-search generation first computed a row. `MAX_PHASE`/`STOP_LOSSES` env vars let the wrapper's single-combo mode (`./scripts/run_v4_backfill_sweep.sh <stop_loss> <close|open_check> [tickers...]`) be scripted тАФ see `scripts/run_backfill_queue.sh` for the ordered multi-step queue (KORU catch-up тЖТ SOXL/KORU low-SL density тЖТ rest of watchlist тЖТ v3-screened non-watchlist tickers), meant to be run directly by the user in their own terminal (tees to `logs/backfill_queue_<timestamp>.log`), not launched by an agent, to avoid `config.json` races with any other in-flight run.
- `scripts/live_sim.py` тАФ manual-step live-sim REPL: drives the real `compute_buy_signal`/`check_sell_condition`/`notify_*` functions against an isolated `cache/trading_sim.db` (via `TRADING_DB_PATH` env override) so the full BUYтЖТplacedтЖТfilledтЖТarmтЖТtrailing-sell Slack sequence can be exercised bar-by-bar without touching the live daemon or `trading_live.db`. `SIM_MODE=1` forces plain-text/typed-input Slack messages (prefixed with a `ЁЯзк SIM MODE`/`ЁЯзк SIM MODE END` header/footer context block, optionally labeled via `SIM_SCENARIO` env var) instead of interactive buttons тАФ the sim never opens its own Socket Mode connection, so real buttons would risk being delivered to the live daemon's connection instead. Real interactive buttons genuinely cannot be tested this way (Slack routes all clicks to whichever process holds the socket, i.e. the live daemon, not the sim) тАФ button *layout* can still be previewed by manually appending a dummy-`action_id` actions block (safe no-op if tapped). `buy`/`sell` REPL commands drive signal checks directly; `pending`/`placed TICKER`/`fill TICKER PRICE`/`remind_buy [--stale]` drive the three-state buy lifecycle (signal тЖТ order placed тЖТ filled) added 2026-07-10. Interactively tested end-to-end with the user 2026-07-09/10 тАФ found and fixed several real bugs in the process (see `docs/conversation_summary.md`).
- `scripts/sim_delayed_sell.py` (2026-07-17) тАФ repeatable, per-ticker/per-node CLI quantifying the cost of intentionally deferring a same-day exit (SL/trailing-stop/TIME) to the next calendar day, the mirror image of the existing `same_day_block` kernel feature (which defers the entry side after a same-day exit, not the exit side of a same-day entry). Uses `scripts/export_trades.py::simulate_trail_both_deferred_sell`, a pure-Python bar-by-bar mirror of `_simulate_trail_both` (entry logic byte-identical to the existing `simulate_trail_both_annotated`) тАФ not a reimplementation of the exit logic from scratch. Only supports `entry_timing='close'` (no `open_check` branch), so results for an `open_check` node (e.g. GDXD) are indicative, not exact. `.venv/bin/python scripts/sim_delayed_sell.py TICKER --window W --z Z --tb TB --arm ARM --sl SL --ts TS --max-hours H`.
- `scripts/sim_chaos_monkey.py` (2026-07-17) тАФ execution-adherence ("chaos monkey") Monte Carlo: quantifies how much of a node's backtested compounded return depends on catching every entry/exit signal exactly on time, the real risk during manual-execution trading (distinct from island/robust-alpha, which both assume perfect adherence). Uses `scripts/export_trades.py::simulate_trail_both_chaos`/`_resolve_miss`, another pure-Python bar-by-bar mirror of `_simulate_trail_both` (verified byte-identical to `simulate_trail_both_annotated` at `miss_rate=0`). Entry signals are only missable at the two daily signal windows (matching the real Slack cadence); exit triggers (SL/trailing-stop/TIME) are missable every bar (matching continuous live monitoring); TP-arming is never missable (internal state change, not a clickable action). Two modes: `drop` (unbounded per-check miss тАФ the opportunity can vanish for good if the underlying condition stops holding first) and `delay` (same coin flip, but capped at `max_delay_checks` consecutive misses of the same still-true condition before it's forced). Runs all `watchlist_id=9` nodes ├Ч both modes ├Ч {1,5,10,20}% miss rates ├Ч 1000 trials by default, writes `output/chaos_monkey_summary.csv`/`chart.png` (gitignored). `.venv/bin/python scripts/sim_chaos_monkey.py [--tickers T...] [--trials N]`. First real run (2026-07-17): 9/12 tickers lose ~15-31% of mean compounded return at a 20% miss rate (KORU worst, -31%); SOXL/DPST are outliers that hold flat or improve тАФ unexplained, flagged in `docs/backlog_cache.md` for follow-up. `drop` vs `delay` track closely everywhere, suggesting the delay cap rarely binds.

## Architecture Notes
- Optimization uses `ProcessPoolExecutor` with up to 10 workers
- Results cached in SQLite to avoid re-running completed nodes
- Multi-generation refinement: macro grid scan тЖТ frontier detection тЖТ fine mesh around top performers
- `active_phase_grid.json` is written by the sweep and read by the Streamlit UI for live progress display
