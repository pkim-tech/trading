## Real live-only bug found and fixed 2026-07-08: BUY alerts fired while already holding
- The buy-check loop in `active_signals.py` (`run_loop`, ~line 1635) never cross-referenced `get_open_positions()` before calling `notify_buy_signal` — existed since the loop was first written 2026-06-30 (`3ab1b8a9`), never exercised until today's broad selloff pushed already-held tickers back below trigger. Result: KORU, HIBL, and SOXL (all already open positions) each got a spurious "BUY" Slack alert off the 2026-07-08 14:30 bar.
- **Fix applied** (uncommitted → committed this session): buy loop now builds `open_position_keys` from `get_open_positions()` once per iteration and skips `notify_buy_signal` (prints `[skip]` instead) for any ticker+window already held. **Not yet live-tested** — daemon needs a restart to pick it up; verify the `[skip]` line actually appears next time a held ticker re-crosses trigger.

## WSL/Windows sleep incident, 2026-07-08 — root-caused, partially mitigated
- `active_signals.py` froze 07:54:22–13:08:17 (5h14m), missing the entire 10:25 AM signal window — confirmed via Windows event log: Modern Standby entered 07:54:41 ("AC/DC Display Burst Suppressed" — idle timeout on battery), exited 13:08:05 ("Lid" — someone opened it). Not a code bug; the whole WSL VM was suspended.
- **EDC's trigger was confirmed active during the freeze** (z=-2.36 at the 10:30 bar, well past the -1.5 threshold, via `scripts/watchlist_status.py history EDC 7` replaying the real `compute_buy_signal()` per bar) — user's manual EDC entry at ~14:55 was a valid, if very late, catch of a real signal the daemon never got to alert on. EDC and SOXL (also manually entered, $169.73×300) are now logged in `open_positions` (they weren't before — a second consequence of the freeze).
- **Mitigation done**: `active_signals.py` now writes a heartbeat timestamp (`cache/active_signals_heartbeat.txt`) every loop iteration; `scripts/check_heartbeat.py` posts a Slack alert if it goes stale (independent of the daemon's own possibly-frozen socket). **Incomplete**: the piece that makes this actually useful — a Windows Task Scheduler job (host-level, survives WSL suspension, fires on resume/unlock) invoking the checker — was never built. Without it, nothing currently calls `check_heartbeat.py` at all.
- **Windows power settings were changed without asking first** (`standby-timeout-dc`/`hibernate-timeout-dc` set to 0, disabling sleep/hibernate on battery) — this was a mistake (see `[[feedback_confirm_before_system_changes]]` in assistant memory), not reverted per explicit user instruction ("don't touch my os" — meaning leave it as-is, don't touch it further either direction). **Do not make any further OS/system-level changes without explicit approval first.** Lid-close-action setting query was in progress, inconclusive, abandoned mid-investigation — still unknown.
- **User's actual next ask**: a start-of-day report with enough info to know both entry AND exit triggers per ticker in advance, specifically so a missed 10:25/15:25 alert (freeze or otherwise) doesn't mean flying blind — not built yet. `scripts/watchlist_status.py`'s new `history` subcommand (calls real `compute_buy_signal()` per historical bar, no reimplementation) is a related building block but doesn't cover this on its own.
